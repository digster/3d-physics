Hello, SDL3

Before any 3D or any physics engine, we need the loop that every interactive program is built on: open a window, and repeatedly handle input, advance the world, and draw. We’ll make a ball bounce to prove it works.

Goal

By the end you will understand the game loop, the idea of delta time, and the three-line “physics” that already lives in a bouncing ball. This is the smallest possible complete program, and every later chapter is a variation on it. The code is chapters/01-hello-sdl3/main.cpp.

The game loop

A game or simulation is not a straight line of code — it’s a loop that runs many times per second. Each pass around the loop produces one frame. The structure never changes:

In SDL3 the skeleton looks like this. Notice there is no “physics engine” yet — just the loop:

SDL_Init(SDL_INIT_VIDEO);
SDL_CreateWindowAndRenderer("Ch1", 900, 600, 0, &window, &renderer);

bool running = true;
while (running) {
    // 1. input: drain the event queue
    SDL_Event e;
    while (SDL_PollEvent(&e))
        if (e.type == SDL_EVENT_QUIT) running = false;

    // 2. update: advance the world by one step (below)
    // 3. draw
    SDL_SetRenderDrawColor(renderer, 26, 28, 36, 255);
    SDL_RenderClear(renderer);
    // ...draw the ball...
    SDL_RenderPresent(renderer);
}

Delta time: making motion frame-rate independent

Here is the subtle part. Different computers run the loop at different speeds — 60, 120, or 144 frames per second. If we moved the ball by a fixed number of pixels per frame, it would fly twice as fast on a 120 fps machine as on a 60 fps one. That’s a bug.

Intuition Speed is “distance per second”, not “distance per frame.” So we measure how long the last frame actually took — call it Δt (delta time) — and move by velocity × Δt. Fast machine, tiny Δt, tiny steps, taken often. Slow machine, bigger Δt, bigger steps, taken rarely. The ball covers the same ground per real-world second either way. 
const Uint64 now = SDL_GetPerformanceCounter();
float dt = float((now - prev) / freq);   // seconds since last frame
prev = now;
if (dt > 0.05f) dt = 0.05f;              // clamp: don't let a hitch teleport the ball

The math: a ball is already physics

Our “world update” is three lines. Gravity is a constant downward acceleration g. Acceleration changes velocity; velocity changes position. In symbols, for one step of length Δt:

vy ← vy+g·Δt y←y+vy·Δt

That’s it — that’s integration, the operation at the core of every simulation in this course. (Updating velocity before using it to move is a deliberate choice called semi-implicit Euler; Chapter 5 explains why it’s the right one.) In code:

vy += gravity * dt;   // acceleration changes velocity
px += vx * dt;        // velocity changes position
py += vy * dt;

Bouncing is just as simple. When the ball crosses a wall, put it back at the wall and flip the velocity component that hit it. Multiplying by a restitution below 1 makes it lose a little energy each bounce, so it eventually settles:

if (py + radius > H) { py = H - radius; vy = -vy * restitution; }  // floor
if (px + radius > W) { px = W - radius; vx = -vx * restitution; }  // right wall
Why flip the sign? A wall can only push away from itself. Reversing the velocity component perpendicular to the wall (and leaving the parallel component alone) is the simplest model of that push. We’ll upgrade this exact idea to 3D contact normals and proper impulses in Part V — but the seed is right here.

Drawing a circle with triangles

SDL has no “draw a filled disc” call. We build one from triangles: a fan of thin wedges around the centre, submitted with SDL_RenderGeometry. That’s worth pausing on, because SDL_RenderGeometry — “here are some colored triangles, fill them” — is the only drawing primitive our 3D renderer will use in Chapter 4. Master it in 2D now and 3D is just “compute better triangle corners.”

// centre vertex, then one vertex per step around the rim, then fan triangles
verts.push_back({ {cx, cy}, color, {0,0} });
for (int i = 0; i <= segments; ++i)
    verts.push_back({ {cx + cos(a)*r, cy + sin(a)*r}, color, {0,0} });
SDL_RenderGeometry(renderer, nullptr, verts.data(), verts.size(),
                   indices.data(), indices.size());

Run it

cmake -B build && cmake --build build -j
./build/bin/ch01_hello_sdl3     # Space relaunches the ball, Esc quits
A crack we’ll fix later This chapter drives the ball with the frame’s delta time. It works, but the simulation subtly changes if the frame rate does — and stacked collisions can jitter. Keep that in the back of your mind: Chapter 5 replaces this with a fixed timestep, and it’s the single most important stability fix in the whole course.

Experiments

  • Set restitution to 1.0. The ball never loses energy — it bounces to the same height forever. Now try 1.05: it gains energy and climbs. (Real contacts are always ≤ 1.)
  • Remove the dt clamp and drag the window around. Without it, a long stall makes the next frame’s Δt huge and the ball jumps through a wall — a first taste of why big time steps are dangerous.
  • Add a horizontal “wind” by nudging vx every frame and watch the arc bend.

Takeaways

  • Every simulation is a loop: input → update → draw.
  • Scale motion by delta time so speed is per-second, not per-frame.
  • “Physics” starts as two lines: acceleration→velocity→position. That operation is called integration.
  • SDL_RenderGeometry (colored triangles) is the one primitive we’ll build the entire 3D renderer on.
