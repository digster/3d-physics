Matrices & Transforms

A matrix is a verb. “Move 5 to the right,” “spin 30°,” “make twice as big” — each is a 4×4 matrix, and multiplying matrices chains those verbs into one. That single idea builds scene hierarchies, positions rigid bodies, and drives the camera.

Goal

Understand what a transform matrix does to a point, why we use 4×4 matrices for a 3D world, and why the order of multiplication matters. Code lives in common/math/mat4.hpp; the demo is chapters/03-matrices/main.cpp.

First: what is a matrix?

A matrix is just a grid of numbers — for us, a 4×4 grid (16 numbers). On its own it’s inert; its meaning comes entirely from one operation: multiplying it by a vector transforms that vector. So when we say “a matrix moves a point,” we mean “multiply the point by the grid of numbers and you get a moved point.”

Two words we’ll use a lot: a column vector is a vector written as a vertical stack of its numbers (that’s the shape matrix multiplication expects). A linear combination (or “blend”) of some vectors just means scaling each one by a number and adding the results — like mixing paints in different amounts. Nothing more exotic than that.

A transform is “new coordinates for a point”

Feed a point in, get a moved point out. The matrix stores the recipe. Multiplying a matrix by a (column) vector produces the transformed vector:

v′=Mv

The cleanest way to read Mv: the result is a blend of the matrix’s columns, weighted by the components of v. The first column is “where the x-axis lands,” the second is “where the y-axis lands,” the third “where the z-axis lands.” A transform is literally a description of where the axes go.

Why 4×4, and the mysterious fourth number

Intuition Rotation and scaling can be done with a 3×3 matrix, but translation cannot — sliding everything over is not a “blend of the axes.” The trick (homogeneous coordinates) is to add a fourth coordinate w and work in 4D. Writing a point as (x,y,z,1) lets a 4×4 matrix fold translation in as ordinary multiplication. A direction uses w=0, which makes translation ignore it — exactly right, since “north” is north no matter where you stand.

Here is the translation matrix. Notice the translation sits in the last column, and the 1 in the point’s w is what pulls it into the result:

[ 100tx 010ty 001tz 0001 ] [xyz1] = [x+txy+tyz+tz1]
Let’s do one by hand

Take the point (2,5,0) and a translation of (10,0,0) (“move 10 to the right”). We write the point as (2,5,0,1) — note the 1 in the w slot. To get the new x, we take the top row of the matrix, (1,0,0,10), multiply it against the point piece-by-piece and add: 1·2+0·5+0·0+10·1=12. The other rows leave y and z alone, so the result is (12,5,0) — moved 10 to the right, exactly as promised. That “multiply a row against the point and add” step is all matrix–vector multiplication ever does; a 4×4 matrix just does it four times, once per output number.

Rotation, in one picture

A 2D rotation by angle θ sends the point (x,y) to (xcosθ−ysinθ,xsinθ+ycosθ). Those coefficients are exactly the rotation matrix. The 3D versions just do this in one of the three coordinate planes at a time.

Composition = multiplication, read right-to-left

Chaining transforms is multiplying their matrices. But the order is the classic trap: in a product, the matrix on the right is applied first.

Order matters because rotation and translation don’t commute: “rotate then move” lands somewhere different from “move then rotate.” This is the most common source of “why is my object in the wrong place” bugs.

The code

Matrix–matrix multiply is just “apply this transform to each column of that one.” It is the whole of composition:

// Column c of the result is M applied to column c of N — i.e. compose them.
inline constexpr Mat4 operator*(const Mat4& m, const Mat4& n) {
    Mat4 r;
    for (int col = 0; col < 4; ++col)
        for (int row = 0; row < 4; ++row) {
            Real sum = 0;
            for (int k = 0; k < 4; ++k) sum += m.at(row, k) * n.at(k, col);
            r.at(row, col) = sum;
        }
    return r;
}

And applying a transform to a point treats it as w=1 so translation is included:

inline constexpr Vec3 transformPoint(const Mat4& m, const Vec3& p) {
    return {
        m.at(0,0)*p.x + m.at(0,1)*p.y + m.at(0,2)*p.z + m.at(0,3),
        m.at(1,0)*p.x + m.at(1,1)*p.y + m.at(1,2)*p.z + m.at(1,3),
        m.at(2,0)*p.x + m.at(2,1)*p.y + m.at(2,2)*p.z + m.at(2,3),
    };
}
Storage note: column-major We store matrices column-by-column (the OpenGL convention), so the translation lives in elements 12–14. The at(row, col) accessor hides that so the code above reads like ordinary math. See the long comment atop mat4.hpp for the full layout.

Run it — a hierarchy from pure composition

./build/bin/ch03_matrices     # press 3 for the orbiting child, W for wireframe

A big cube spins in place. Press 3 and a small cube orbits it. The orbit isn’t special-cased — it’s built entirely by composing transforms:

Mat4 orbit =
    rotationY(t * 1.1f) *            // revolve around the big cube (acts last)
    translation({3, 0, 0}) *         // push out along +x (the arm)
    rotationY(t * 2.5f) *            // the child's own spin
    rotationX(t * 1.3f);             // (acts first, in the child's local space)

Read it bottom-to-top: the child spins in its own space, is pushed out along an arm, and then that whole arm is rotated about the origin — which sweeps the child around in a circle. That’s how a moon orbits a planet, or a hand rides on a forearm.

Experiments

  • Swap the order of the translation and the first rotationY in orbit. The child stops orbiting and just spins in place off-center — proof that order changes meaning.
  • In scene 2 the cube also pulses in scale. Change spin * scaling(...) to scaling(...) * spin and watch the box shear as it turns — you’re now scaling after rotating.
  • Press W to see the wireframe and confirm the cube is a genuine 3D box being transformed, not a flat picture.

Takeaways

  • A matrix transforms a point; its columns say where the axes land.
  • 4×4 + a w coordinate lets one matrix do translation and perspective too.
  • Compose transforms by multiplying; the rightmost is applied first.
  • Hierarchies (and, soon, camera and rigid bodies) are just transform chains.
