Numerical Integration

Every simulation moves forward in time by small steps. How you take that step decides whether your world is stable or slowly explodes. This is the most important chapter in Part I — get integration right and the rest of the engine has solid ground to stand on.

Goal

Understand what it means to “integrate” motion, why the naive method leaks energy, and why one tiny reordering fixes it. Then meet RK4, and learn the fixed-timestep loop that makes the whole engine deterministic. Code: chapters/05-integration/main.cpp.

What we’re actually doing

Physics gives us acceleration (from forces: a=F/m). We want position. Acceleration changes velocity; velocity changes position. But we can’t solve for all time at once — we take a small step Δt and repeat. That stepping is numerical integration, and there’s more than one way to do it.

Explicit Euler — simple, and subtly wrong

The obvious approach: use the current velocity to move, then update the velocity.

x′=x+v Δt v′=v+a Δt
The problem Explicit Euler uses the old velocity for the whole step, always a little behind where the object really is. On anything that oscillates — a spring, an orbit, a bouncing stack — that lag consistently adds energy. The orbit spirals outward; the spring shakes itself apart. It doesn’t just make errors, it makes errors that always push the same way.

Semi-implicit (symplectic) Euler — the one-line fix

Update the velocity first, then move with the new velocity:

v′=v+a Δt x′=x+v′ Δt
Intuition That’s the only change — two lines swapped. But now the errors alternate sign instead of accumulating, so total energy wobbles within a tight band instead of drifting. It’s not perfectly accurate, but it’s stable, which matters far more for games and interactive physics. This is what almost every game engine uses, and what we’ll use throughout the course.

RK4 — spend four samples for accuracy

When you need accuracy (say, a stable orbit over long times), Runge–Kutta 4 takes four “trial” slope measurements across the step and averages them with clever weights:

y′=y+ Δt6 (k1+2k2+2k3+k4)

It’s roughly four times the work per step and much more accurate, but — crucially — it is not automatically stable for stiff systems, and it doesn’t conserve energy exactly either. For real-time rigid-body physics, cheap-and-stable (semi-implicit Euler) usually beats accurate-and-expensive (RK4). Knowing both, and why we pick the humble one, is the point.

Also worth knowing: Verlet Position Verlet stores two past positions instead of a velocity and infers motion from their difference. It’s symplectic like semi-implicit Euler and makes position constraints trivial — which is exactly why Part II’s ropes and cloth will reach for it.

The code

Here are the two Eulers side by side. The entire difference — stable vs. ever-growing — is the order of two lines:

State stepExplicitEuler(State s, const AccelFn& a, Real dt) {
    const Vec3 acc = a(s.pos, s.vel);
    s.pos += s.vel * dt;   // position uses the OLD velocity...
    s.vel += acc * dt;     // ...then velocity updates. This ordering leaks energy.
    return s;
}

State stepSemiImplicitEuler(State s, const AccelFn& a, Real dt) {
    const Vec3 acc = a(s.pos, s.vel);
    s.vel += acc * dt;     // update velocity FIRST...
    s.pos += s.vel * dt;   // ...then move with the NEW velocity. Symplectic.
    return s;
}

The other half: a fixed timestep

Chapter 1 moved the ball by the frame’s delta time. That makes the physics depend on frame rate — a slow frame takes one giant, unstable step. The fix is to integrate at a fixed Δt (we use 1/120 s), accumulating real time and spending it in fixed chunks:

accumulator += frameTime;          // real time elapsed this frame (clamped)
while (accumulator >= dt) {
    fixedUpdate(dt);               // one deterministic step, always the same size
    accumulator -= dt;
}
Real alpha = accumulator / dt;     // leftover fraction → interpolate when drawing
Why fixed matters here specifically Integrator stability, restitution, and stacking all depend on a constant step. A fixed step also makes the sim deterministic — run it twice, get the same result — which is what makes the --selftest harness and reproducible bug reports possible. This is Glenn Fiedler’s “Fix Your Timestep,” and it’s in common/app.cpp.

Experiments

  • In scene 1, watch the HUD: explicit Euler’s energy E only ever climbs, while the other two hold steady. That upward number is the spiral.
  • In the widget, set Δt to its maximum and hit reset — coral diverges almost immediately while teal and lime survive. Stability is a property of the method and the step size together.
  • In scene 2, note how well all three track constant-acceleration motion. Integrators only diverge on velocity-dependent forces (springs, orbits) — which is most of interesting physics.

Takeaways

  • Integration steps a simulation forward; the method matters as much as the math.
  • Explicit Euler leaks energy; semi-implicit Euler fixes it by updating velocity before position.
  • RK4 is accurate but not automatically stable; real-time physics prefers stable.
  • Integrate at a fixed timestep for stability and determinism.
