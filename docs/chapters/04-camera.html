Camera & Projection

How does a scene full of 3D points become a flat image on your screen? Through a short pipeline of matrix multiplies and one magic division. By the end of this chapter you’ll have written a complete software 3D renderer — the thing that lets us see every simulation to come.

Goal

Follow a single vertex from the model all the way to a pixel, and understand each stage: the view matrix, the perspective matrix, the perspective divide, backface culling, and the painter’s algorithm. Code: common/render/renderer3d.cpp and common/render/camera.hpp.

The pipeline

The view matrix: move the world, not the camera

Intuition There is no real “camera” in the math. Instead of moving a camera to look at the scene, we move the whole scene so that the camera would sit at the origin looking down one axis. The view matrix is that world-shifting transform. It’s built from the camera’s position and the direction it faces.

Our lookAt builds three axes — right, up, and forward — from the eye position and target, and packs them into a matrix. The demo’s OrbitCamera keeps things friendly: it stores a target, a distance, and two angles (yaw = how far it has swung left/right around the target, pitch = how far up or down it is tilted), and reconstructs the eye position from them.

“Spherical coordinates”, in plain terms

Describing the camera by two angles and a distance is exactly how you’d point at a star: an azimuth (which compass direction), an elevation (how high above the horizon), and how far away. That’s a spherical coordinate. The position() code below just converts those three friendly numbers back into ordinary (x,y,z) coordinates using sine and cosine — that conversion is all “spherical-to-Cartesian” means. Dragging the mouse changes the angles; the wheel changes the distance.

Vec3 position() const {                 // spherical → Cartesian
    const Real cp = std::cos(pitch), sp = std::sin(pitch);
    const Real cy = std::cos(yaw),   sy = std::sin(yaw);
    return { target.x + distance*cp*sy,
             target.y + distance*sp,
             target.z + distance*cp*cy };
}
Mat4 viewMatrix() const { return lookAt(position(), target, axis::Y); }

Perspective: why distant things look small

The perspective matrix is the lens. Its job is to copy the camera-space depth into the w coordinate, so that the next step — dividing by w — shrinks far things. That division is the entire secret of perspective:

(x,y,z,w) ⟶ (xw,yw,zw)
Why this makes depth The perspective matrix sets w=−zview — the distance in front of the camera. So a point twice as far away has twice the w, and dividing by it halves the point’s x and y on screen. Two objects the same real size, one twice as far, draw at half the size. That’s literally similar triangles, done by one divide.

Field of view is baked into this matrix. A narrow FOV looks like a telephoto lens (flat, compressed depth); a wide FOV looks fish-eye (exaggerated depth). The demo binds it to the [ and ] keys so you can feel the difference on a fixed scene.

Two cleanups before drawing

Backface culling

On a closed solid, you never see the back of a face — the front of the object is in the way. So for each triangle we ask “does this face point toward the camera?” with a dot product, and skip it if not. That halves the triangles we draw, for free:

// The view-space normal points toward the camera for front faces, so a
// non-negative dot with the triangle centroid means "facing away" → skip it.
const Vec3 viewNormal = cross(v1 - v0, v2 - v0);
const Vec3 centroid   = (v0 + v1 + v2) * (1.0f / 3.0f);
if (!wireframe_ && dot(viewNormal, centroid) >= 0) continue;

The painter’s algorithm

We have no depth buffer, so how do near objects hide far ones? We sort every triangle back-to-front and draw them in that order — like a painter laying down the background before the foreground. The key design choice: the renderer collects triangles from all meshes into one list and sorts once, so different objects occlude each other correctly.

Its one weakness Triangles that interpenetrate can’t be ordered correctly by a single depth value, so they may flicker for a frame. For a physics tutorial that’s fine — interpenetration is exactly the state the collision solver is about to fix — and the W wireframe toggle makes any overlap easy to read.

Flat shading, in one line of light

To give shapes form we tint each face by how squarely it faces a light. A face pointing straight at the light is fully bright; one turned edge-on gets nothing. “How squarely does it face the light?” is a question the dot product from Chapter 2 answers, so the whole shading model is one line:

brightness=max(0,n·ℓ)

Here n is the face’s normal — the arrow sticking straight out of the surface, which we get as the cross product of two of the triangle’s edges — and ℓ is the direction toward the light. Their dot product is 1 when the face aims right at the light and 0 when it faces sideways; the max(0,…) clamps away negatives (a face pointing away from the light just gets no light, not negative light). This is called a Lambert term, after the simplest physical model of a matte surface. We add a small ambient floor so faces in shadow are dim rather than pure black.

./build/bin/ch04_camera       # drag to orbit, wheel to zoom, [ ] change FOV, 1-3 scenes

Fly around the primitive zoo. Scene 2 is a grid of cubes — a good stress test for the depth sort; watch how the cubes always occlude one another correctly as you orbit.

Experiments

  • Press ] repeatedly to widen the FOV toward fish-eye, then [ to a narrow telephoto. Same scene, dramatically different sense of depth — all from one matrix.
  • Orbit until you’re nearly edge-on to the ground plane. The near-plane clipping keeps geometry that crosses the camera from exploding into garbage.
  • Toggle W: with culling off in wireframe mode you can see the back faces you normally never render.

Takeaways

  • Rendering is a matrix pipeline: model → view → clip, then ÷ w to a screen.
  • The view matrix moves the world in front of a camera fixed at the origin.
  • The perspective divide is what makes distant things small.
  • No depth buffer? Cull backfaces and sort back-to-front.
