Broadphase

Before we can make things bounce, we have to know what’s touching what. Checking every pair of bodies is hopeless for a big scene — it grows with the square of the count. Broadphase is the cheap first pass that throws away the pairs that are obviously nowhere near each other, so the expensive exact tests only run on a handful of real candidates.

Goal

Understand why all-pairs testing doesn’t scale, and three ways to avoid it using loose bounding boxes. Engine: physics/aabb.hpp, physics/broadphase.hpp. Demo: chapters/12-broadphase/main.cpp.

The O(N²) problem

With N bodies there are N(N−1)2 possible pairs. That’s fine for 10 bodies (45 pairs) but brutal for 1000 (about half a million) — every frame. We say this scales as O(N2) (“order N squared”: double the bodies, quadruple the work). Most of those pairs are a body on the left of the room and one on the right — never touching. Broadphase’s job is to skip them almost for free.

The tool: axis-aligned bounding boxes

Intuition An AABB (axis-aligned bounding box) is the smallest world-axis-aligned box that fully wraps a shape — just two corners, a min and a max. Two AABBs can only overlap if their shadows overlap on all three axes at once, which is a handful of number comparisons. If even one axis has a gap, the boxes (and the shapes inside them) are apart. So we test cheap AABBs first, and only bother with the real shape test when their boxes overlap.

Three ways to find the pairs

1 · Brute force — the honest baseline

Test all pairs’ AABBs directly. It’s O(N2) and correct — we keep it as the yardstick the clever methods must match.

2 · Sweep-and-prune

Sort the boxes by their minimum x. Then sweep a line left to right: keep an “active” set of boxes whose x-span you’re still inside, and only compare against those. Once a box’s x ends, it can never overlap anything further right, so drop it. Bodies far apart in x are never even compared.

3 · Uniform grid

Chop space into a grid of cells and drop each box into the cells it covers. Two boxes can only touch if they share a cell, so we only compare within-cell boxes. It’s wonderfully simple and fast when bodies are roughly the same size.

Same answer, far less work

The important guarantee: all three return the same set of candidate pairs — a faster method that disagreed with brute force would be a silent bug (the unit tests check exactly this on a random cloud). The clever ones just reach that answer with a fraction of the comparisons.

Run it

./build/bin/ch12_broadphase     # 1 uniform grid · 2 sweep & prune · 3 brute force

A cloud of 140 spheres drifts in a box. Amber links join candidate pairs; coral marks the ones actually overlapping. Read the HUD: the broadphase flags only a few dozen candidates out of ~9700 possible pairs — and finds the exact same overlaps brute force does.

Experiments

  • Switch between scenes 1–3. The candidate count and the coral overlaps are identical every frame — the methods agree, they just work harder or less.
  • In the code, drop the grid’s cellSize very small or very large. Too small wastes memory on empty cells; too large dumps everything into one cell and degrades back toward O(N2). The sweet spot is roughly a typical body’s size.

Takeaways

  • All-pairs testing is O(N2) and doesn’t scale.
  • AABBs overlap only if they overlap on all three axes — a few cheap comparisons.
  • Sweep-and-prune and the uniform grid both skip far-apart pairs; all methods agree on the result.
  • Broadphase hands a short candidate list to the exact narrowphase (next chapter).
